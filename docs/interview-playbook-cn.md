## 1. 这个项目到底是干嘛的？

项目名：MarketCopy Agent

中文解释：

这是一个面向英文商品 Listing 生成场景的 AIGC / Agent 产品助理项目。它把卖家的真实需求拆成三个 Agent 阶段：

1. Style Clone Agent：分析历史爆款英文文案的风格。
2. Listing Generation Agent：根据新品信息生成英文标题、卖点 bullet、描述。
3. Risk Check Agent：检查夸大、绝对化、可能有风险的表达，并给出更安全的替换建议。

你可以这样讲：

> 这个项目不是为了证明我很懂跨境电商，而是展示我能从真实用户访谈里抽象需求，把一个模糊业务问题拆成可运行的 AIGC Agent 工作流，并考虑模型选择、成本控制、API 失败兜底和面试演示稳定性。

## 2. 为什么它对应 AIGC / Agent 产品助理岗位？

AIGC / Agent 产品助理面试官通常想看这些能力：

- 你能不能从真实需求里拆出 AI 工作流。
- 你知不知道大模型不是一个按钮，而是要设计输入、输出、步骤和兜底。
- 你能不能把 Prompt、模型、API、成本、稳定性这些东西讲清楚。
- 你有没有实际跑起来的 Demo，而不是只写概念。
- 你能不能解释为什么这么设计，而不是只说“让 AI 生成一下”。

这个项目能对应这些点：

- 需求来源：来自一个做亚马逊的朋友的真实需求访谈。
- 产品拆解：把“帮我写英文 Listing”拆成风格分析、文案生成、风险检查。
- Agent 思路：不是单次生成，而是分阶段处理任务。
- 模型策略：支持 Mock、Gemini、DeepSeek，可选模型。
- 成本意识：默认 Mock 不花 token，Gemini 免费层优先，单次调用返回完整结构。
- 稳定性：API key 没配或者接口失败时，自动降级 Mock，不影响演示。
- 工程落地：React + TypeScript + Vite 做了可运行界面，GitHub 已同步。

## 4. 页面上每个区域是什么意思？

### 顶部标题区

你会看到：

```text
MarketCopy Agent
基于卖家访谈的 AIGC Agent 工作流：Style Clone → Listing Generation → Risk Check，输出英文商品文案。
```

面试说法：

> 这里我把项目定位直接写在首屏，面试官一打开就能看到它不是普通文案生成器，而是一个基于卖家访谈的三阶段 Agent 工作流。

### Model 下拉框

当前有四个选项：

- Mock Demo：默认选项，不花 token，最适合面试演示。
- Gemini 3.1 Flash-Lite：免费层优先，适合低成本 API 演示。
- Gemini 3.5 Flash：能力更强一点，适合质量优先的场景。
- DeepSeek Chat：低成本备选模型。

面试说法：

> 我没有把模型写死，而是做成可选模型。实际产品里不同用户可能更关心成本、速度或质量，所以这里保留了模型选择能力。默认 Mock 是为了保证演示稳定，也避免每次演示都花 token。

### 左侧 Seller Context

这里是输入区。

字段说明：

- New product：新品名称，比如 AeroFit Wireless Earbuds。
- Selling points：新品卖点，比如长续航、防水、佩戴舒适。
- Keywords：关键词，比如 wireless earbuds、noise cancelling。
- Desired tone：希望的语气，比如 premium but practical。
- Historical listing 1/2/3：三条历史英文商品文案，用来学习风格。

面试说法：

> 这个输入结构来自卖家真实工作流。卖家不是凭空写新品文案，通常会参考过去表现好的 Listing，所以我让用户输入三条历史文案作为风格参考。

### 右侧 Agent Stages

这里展示三个 Agent 阶段：

1. Style Clone Agent
2. Listing Generation Agent
3. Risk Check Agent

面试说法：

> 我把大任务拆成三个小任务，是为了让过程更可解释。面试官和用户都能看懂 AI 不是随便生成，而是先分析风格，再生成文案，最后做风险检查。

### 底部结果区

生成后会出现三块结果：

- Style Analysis / 风格分析
- Generated Listing / 英文文案
- Risk Check / 风险检查

每块都有 Copy 按钮，可以复制结果。

面试说法：

> 结果不是只给一段文案，而是保留了分析过程和风险提醒，这样更像产品工具，不只是 ChatGPT 对话。

## 9. 面试官可能追问：你这样答

### Q1：你没做过跨境电商，为什么做这个项目？

答：

> 我确实不是跨境电商专家，所以我没有把项目包装成行业专家系统。我的切入点是 AIGC 产品助理能力：从真实访谈里抽象需求，把模糊需求拆成可运行的 AI 工作流。跨境电商只是业务场景，核心展示的是需求拆解、Agent 流程设计、模型选择和成本控制。

### Q2：这里的 Agent 是什么意思？是真正自主 Agent 吗？

答：

> 我这里的 Agent 更偏产品工作流意义，不是完全自主行动的复杂 Agent。它把一个大任务拆成多个有明确职责的阶段：风格分析、文案生成、风险检查。每个阶段有自己的输入输出，过程可解释，适合 MVP 阶段展示。

### Q3：为什么不直接让 ChatGPT 一次性生成？

答：

> 一次性生成也能出结果，但不可解释，也不容易产品化。我拆成三个阶段，是为了让用户理解 AI 怎么得出结果，也方便后续优化每个模块，比如单独优化风险检查或者风格分析。

### Q4：为什么要 Mock？

答：

> Mock 是为了演示稳定和低成本。面试或者早期验证时，网络、API key、额度都可能不稳定。Mock 能保证主流程随时可跑，而且数据结构和真实 API 返回保持一致，后续替换真实模型成本很低。

### Q5：API 很简单吧？这个项目难点在哪里？

答：

> API 调用本身不复杂，真正重要的是产品设计：用户输入怎么组织、任务怎么拆、输出结构怎么定义、模型失败怎么兜底、成本怎么控制、结果怎么让用户看懂。这个项目重点不是炫 API，而是把 AI 能力做成可用的产品流程。

### Q6：你怎么控制 token 成本？

答：

> 第一，默认 Mock 不消耗 token。第二，真实 API 模型优先选择 Gemini Flash-Lite 这种免费层/低成本模型。第三，历史 Listing 输入限制长度。第四，Prompt 要求返回紧凑 JSON。第五，一次模型调用返回三个模块结果，避免多次调用。

### Q7：为什么要支持多个模型？

答：

> 因为真实产品里不同用户会有不同诉求：有的人要低成本，有的人要质量，有的人要稳定。模型选择能体现产品的灵活性，也方便后续做 A/B 测试或按套餐分层。

### Q8：API key 安全怎么处理？

答：

> API key 不放在前端代码里，而是放在服务端环境变量里。前端只把输入和模型选择发给后端，后端再调用模型 API。这样不会把密钥暴露给浏览器用户。

### Q9：风险检查准确吗？

答：

> 现在是 MVP 阶段，风险检查分两层：本地规则先检查明显高风险词，比如 best、100%、guaranteed；真实模型也可以进一步给出表达建议。它不是法律或平台合规保证，而是一个辅助提醒模块。

### Q10：如果继续迭代，你会怎么做？

答：

> 我会优先做四件事：第一，加用户上传历史 Listing 或店铺风格库；第二，加生成结果评分和人工反馈；第三，加不同平台的风险词库，比如 Amazon、Shopify、TikTok Shop；第四，加导出和版本管理，让卖家能对比不同模型结果。

答：

> 可以看几个指标：生成一次 Listing 节省的时间、用户复制率、用户修改率、风险词命中率、用户对生成结果的满意度。如果是真实业务，还可以看发布后的点击率和转化率变化。

## 11. 项目里的技术词怎么解释？

### AIGC

AI Generated Content，AI 生成内容。这里生成的是英文商品 Listing。

### Agent

这里不是神秘概念，可以理解成“有明确职责的 AI 工作节点”。本项目有三个节点：分析风格、生成文案、检查风险。

### Listing

电商商品详情页里的标题、卖点、描述等内容。亚马逊里经常叫 Listing。

### Prompt

给模型的任务说明。好的 Prompt 不只是问一句话，而是规定输入、输出格式、语气和约束。

### Token

模型计费和处理文本的基本单位，可以粗略理解成字和词的片段。输入越长、输出越长，token 越多，成本越高。

### Mock

模拟数据。没有真实调用模型，但返回和真实模型类似的结果，用来保证演示稳定。

### API

程序调用模型的接口。前端把用户输入传给后端，后端再调用 Gemini 或 DeepSeek。

### Fallback

兜底策略。API 失败时，不让用户看到报错，而是自动返回 Mock 结果，让流程继续跑。

### JSON

一种结构化数据格式。让模型按固定结构返回，前端才能稳定展示。

## 15. 面试时最稳的完整话术

你可以几乎照读：

> 我先简单介绍一下这个项目。MarketCopy Agent 是一个 AIGC / Agent 产品助理 MVP，场景是英文商品 Listing 生成。这个需求来自我和一个做亚马逊的朋友的交流。她的痛点不是单纯写英文，而是新品文案要参考历史高转化 Listing 的风格，同时融合新品卖点和关键词，还要避免夸大或绝对化表达。
>
> 所以我把它拆成了三个 Agent 阶段。第一步 Style Clone Agent 分析历史文案的语气、标题结构和销售逻辑；第二步 Listing Generation Agent 生成新品标题、五条 bullet 和描述；第三步 Risk Check Agent 检查风险表达并给出更安全的替换建议。
>
> 这里我默认用 Mock Demo 演示，因为面试环境里 API key 和网络不一定稳定，而且 Mock 不花 token。真实 API 也预留了 Gemini 和 DeepSeek 模型选择，API key 放在服务端，前端不会暴露。如果接口失败，系统会自动 fallback 到 Mock，保证演示流程不中断。
>
> 这个项目主要想体现的是我对 AIGC 产品落地的理解：不是简单调用模型，而是从用户需求出发，设计输入输出、拆分工作流、控制成本、处理异常，并把它做成一个可以运行和展示的 MVP。
